自宅NASにClaude Codeを導入して、
スマホからも動かせる「創作基地」を作った日

🟢 やさしい版（はじめての人へ）

専門用語は出てきますが、覚える必要はありません。出てきた順に「＝かんたんに言うと何か」を、そのつど付けていきます。

今日やったこと（ひとことで）

家に置いた「画面もキーボードも無いPC」（NAS）の中でAIを動かし、手元のパソコンやスマホから遠隔で呼び出せるようにして、接続を切ってもAIが動き続け、さらに両方のパソコンでファイルが自動で同じにそろう——そんな“創作の基地”を作った一日です。

まず「NAS」って何？

NAS＝マウスもキーボードもモニターも無いPC、と思ってください。中身はふつうのパソコンと同じですが、画面をつながずに電源を入れっぱなしにしておける“留守番PC”です。だから、置きっぱなしで働かせておけます。

画面もキーボードも無いのに、どうやって動かすの？

ここで SSH という接続方法を使います。

SSHとは、かんたんに言うと「手元のパソコンから、離れたPC（NAS）の操作画面を呼び出して、安全な回線でそのまま操作できる仕組み」です。手元のパソコンで打った命令が、そのままNASに届いて動くので、画面が無くても困りません。

その留守番PC（NAS）に、AI（Claude Code）を入れる

SSHでNASにつないだら、その中にAIの相棒（Claude Code）をインストール。これで、手元のパソコンからNASの中のAIに指示を出せるようになりました。

スマホからも呼べるようにした

さらに、スマホのアプリからも同じAIにつなげるよう設定。外出先からでも、NASの中のAIに指示を出せます。

最初の“困った”：接続を切ると、AIも止まる

ところがSSHの接続には、「切ると、その中で動かしていた作業も一緒に終わってしまう」という性質があります。パソコンを閉じたり接続を切ったりすると、せっかく動かしていたAIの作業まで止まってしまう。これが最初の状態でした。

解決：tmux で「閉じても消えない作業部屋」をNASの中に作る

tmuxとは、かんたんに言うと「NASの中に、“閉じても消えない作業部屋”を用意しておく仕組み」です。AIをこの作業部屋の中で動かすと、手元の接続を切っても、部屋の中では作業が続きます。次に手元のパソコンやスマホからつなぎ直せば、その作業部屋の続きがそのまま出てきます。

コツは「操作する接続」と「作業そのもの」を切り離すこと。作業をNAS側の作業部屋（tmux）に置いておけば、手元の接続は好きなときに切ったりつないだりできます。これで“どこにいてもAIを呼べる”状態になりました。

手元のPCとNASで、ファイルを自動でそっくり同じに（Syncthing）

Syncthingとは、かんたんに言うと「2台のパソコンの“決めたフォルダ”を、つねに同じ中身に保ってくれる自動同期の仕組み」です。手元のパソコンで描いても、NASの中のAIが書いても、両方のフォルダがいつも同じ中身になります。実際にマンガ一式（約1.5GB）が、まるっと両方でそろいました。

今日の気持ち

はじめての言葉ばかりでしたが、一つずつ「＝かんたんに言うと何か」を確かめながら進めたら、意外とすんなり分かりました。NASは画面のないPC、SSHは遠隔操作の接続、tmuxは閉じても消えない作業部屋——正体をやさしい言葉に置きかえれば、むずかしいものも怖くない。そう実感できた一日でした。

🔧 くわしい版（用語そのまま）

今日の活動テーマ

  • 自宅のUGREEN NAS（DXPシリーズ）に、Claude Code（ターミナルで動くAIの相棒）を導入する
  • スマホのClaudeアプリから、NAS上のClaudeを操作できるようにする
  • PCを閉じてもNAS上のClaudeが動き続ける「常時起動」を用意する
  • ローカルPCとNASで、マンガ制作フォルダ（manga-production）を自動同期する土台をつくる

今日やったこと（コマンド再現）

実際の端末画面ではなく、打ったコマンドを再現ブロックで載せます（生のスクリーンショットは使いません）。

1. NASにSSH接続

WindowsのPowerShellから、モニターの無いNASにログインして遠隔操作します。住所は家の中だけで通じる私設IP（192.168.x.x の形）。

▷ PowerShell（PC）
PS> ssh 管理者アカウント@192.168.x.x # NAS（Linux）のシェルに入る

2. Claude Code の導入

NAS上でインストールし、置き場所 ~/.local/bin をPATHに登録。claude の一言で起動できるように。

▷ NAS（Linux シェル）
$ curl -fsSL https://claude.ai/install.sh | bash $ export PATH="$HOME/.local/bin:$PATH" $ claude # AIの相棒を起動

3. スマホから操作（Remote Control）

Claude内で /remote-control を有効化。スマホのClaudeアプリから同じセッションに接続でき、処理はNASで動き続けます（スマホは“リモコン”）。

▷ Claude 内
> /remote-control # 表示された案内からスマホのClaudeアプリで接続（接続URLは非公開）

4. 常時起動（tmux）

SSHの接続には「切ると、その中で動かしていた作業も一緒に終わる」性質があります。tmux（＝NASの中に用意する“閉じても消えない作業部屋”）の中でClaudeを動かすと、操作する接続と、作業そのものが分離され、接続を切ってもNAS上で動き続けます。

▷ NAS（Linux シェル）
$ sudo apt install tmux $ tmux # 消えない作業部屋の中へ $ claude # この中で起動 → 切っても動き続ける # 離脱: Ctrl+b → d ／ 復帰: tmux attach

5. Docker＋Syncthing（NAS側）

UGOSのDockerで linuxserver/syncthing のコンテナを作成。GUI（管理画面）で設定しました。

🐳 Docker コンテナ設定UGOS 管理画面（GUI）再現
イメージlinuxserver/syncthing
権限（PUID/PGID）自分のユーザーIDに合わせる
ポート8384（設定画面） / 22000（同期通信）
フォルダ/data（同期対象を割り当て）

6. Syncthing（PC側）とペアリング

▷ PowerShell（PC）
PS> winget install GermanCoding.SyncTrayzor
🔗 デバイス ペアリング＆同期Syncthing 画面（GUI）再現
ペア方法お互いのデバイスIDを登録（IDは非公開）
同期フォルダmanga-production（約1.5GB／3,245ファイル）
結果PC→NAS へ完全同期。両方「最新」を確認
おまけ：ソフトを入れる“調達係（パッケージマネージャー）”は、Linuxは apt、Windowsは winget。OSが違うだけで役割は同じ（sudo apt install 〇〇 ≒ winget install 〇〇）。

学び（やさしい説明）

NASとは

NAS（Network Attached Storage）は「家に置く、電源をつけっぱなしにできる小さなサーバー」。中身はふつうのPCと同じで、モニターやキーボードを普段つながないため、操作は別のPCから遠隔で行う。SSH（Secure Shell）は「離れたコンピュータのコマンド画面を、暗号化された安全な回線で操作する方法」。住所は家の中だけで通じる私設IP（192.168.x.x の形）。

CLIとGUIとは

  • CLI（Command Line Interface）：黒い画面に文字でコマンドを打つ方式。細かい作業・自動化に強い。今日のNAS操作やインストールはこれ。
  • GUI（Graphical User Interface）：アイコンやボタンをマウスで操作する方式。直感的。UGOSの管理画面やSyncthingの設定画面はこれ。
  • 同じNASを「文字で命令する（CLI）」か「画面を触る（GUI）」かの違い。今日は両方を行き来した。

SyncthingとDocker

Syncthingは「2台の機器のフォルダを、つねに同じ中身に保ち続ける自動同期ツール」。クラウドを介さず、機器同士が直接・暗号化してやり取りする（P2P）。片方が壊れても、もう片方に全データが残る。NASは常時稼働、PCは起動したときに差分を自動で追いつく。Dockerは「本体を汚さず、隔離した小部屋（コンテナ）でアプリを動かす仕組み」。イメージ（設計図＝カセット）からコンテナ（動く実体）を起動し、ポート（番号付き端子）で設定画面(8384)やデータ(22000)につなぐ。コンテナのPUID/PGIDを自分の番号にそろえ、同期ファイルをNAS上のClaudeも読み書きできるようにした。

活動を経てのジャーナリング

今日はNASという「自分の基地」を、遠隔から一日かけて立ち上げた。いちばん腑に落ちたのは、『操作する接続』と『作業そのもの』を分けるという発想だった。SSHの接続に作業がぶら下がっていると、線を切った瞬間にすべて止まる。でもtmuxで作業をNAS側に置いてしまえば、操作する側（PC・スマホ）は好きに抜き挿しできる。これは道具の話にとどまらず、『動かし続けたいものは、目の前の自分の接続に依存させない』という設計の考え方そのものだと感じた。

もうひとつは、むずかしい用語も“正体をやさしい言葉に置きかえる”と一気に分かるということ。NASは画面のないPC、SSHは遠隔操作の接続、tmuxは閉じても消えない作業部屋——一つずつ「＝かんたんに言うと何か」を確かめたら、初めての概念が驚くほどすっと入った。ただしその言い換えは、最後まで筋道が通っていて初めて効く。無理な例え（今回でいえばNASを"独立したゲーム機"と呼ぶような）は、かえって混乱を生むと痛感した。

最後に、今日作ったのは『マンガを描く仕組み』そのものではなく『どこにいても創作を進められる土台』だということ。家のNASが常に起き、スマホから指示でき、PCと自動で同期する——場所と時間の制約が一枚はがれた感覚がある。次は、この土台の上で実際に4話を動かしてみたい。

次にやること

  • スマホからNASのClaudeで「4話の原案」を作成させる（未作成分）。
  • 予約実行の設計：Claude Codeの /loop、またはNASの cron ＋ claude -p（ヘッドレス）で、決めた時間にNASのClaudeを自動実行できるようにする。
  • PC側Syncthingに任意でGUIパスワードを設定する。
  • 「オープンソース」ページに、今日の子トピック（NASとは／CLIとGUIとは／Syncthingとは）を再利用リンクで並べる。
