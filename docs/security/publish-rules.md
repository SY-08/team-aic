# 公開ルールの正本（PUBLISH RULES / 非公開ルール）

> このファイルは team AIC のホームページ公開における**最重要ルールの正本**です。
> 「公開前チェック（関所）」はこのファイルを参照して、成果物を検査・自動修復します。
> このルール自体は運用の指針であり、ここに実際の機微情報（IP・鍵など）を書いてはいけません。

## 1. 絶対に公開しない情報（検出→自動修復の対象）

関所は Notion→ビルド→GitHub Pages の**公開直前**に、生成された成果物（HTML/JS/JSON/画像）を検査する。
機微情報を検出しても**止めない**。安全側に倒して**自動修復して通す**。直した内容はログに残す（公開はログを待たない）。

| 種別 | 例（値は各自の実値） | 修復方法 |
|---|---|---|
| 私設IPアドレス | 192.168.系 / 10.系 / 172.16〜31.系 | 伏字化：`192.168.x.x` |
| 機器の固有デバイスID | Syncthing Device ID 等の長い英数-区切り | 伏字化：`XXXXXXX-…-XXXXXXX` |
| 接続URL（遠隔操作リンク） | `claude.ai/code/session_…` 等 | リンク削除／`（接続URLは非公開）` に置換 |
| OSユーザー名・ホスト名 | ログイン名・端末名 | 一般名に置換：`user` / `host` |
| 内部パス | `/home/ユーザー名/…` / `C:\Users\ユーザー名\…` | 一般化：`/home/user/…` / `C:\Users\user\…` |
| APIキー・トークン | `sk-…` `ghp_…` `AIza…` `xox…` 等 | 完全削除（伏字ではなく除去） |
| 生スクリーンショット | 端末やGUIの本物の画面写真 | 該当箇所をマスク／「コマンド付き再現ブロック」「再現カード（クリーン図解）」へ差し替え |
| 個人情報・第三者特定情報 | 住所・電話・家族/顧客/取引先名・守秘義務情報 | 削除／一般化 |
| 税務・医療・法律の断定表現／未確認情報 | — | 断定を避ける表現へ／未確認は「未確認」と明記 |

## 2. ホームページ共通ルール（表現）

- **論理構造を壊さない**：比喩で具体化するときも、筋道が通らない例えは使わない。一つの言い換えを使うなら最後まで一貫させ、**破綻するくらいなら比喩を捨てて正確な説明に戻す**（無理に統一しない）。
- **原則：一つの大きなたとえで全部を縛らない。** まず**正確でやさしい言い換え**（例：NAS＝画面のないPC）を土台にし、専門用語が出るたびに**「◯◯とは、かんたんに言うと〜」を都度そえる**。図解は必ずセットで、色・アイコンで見やすく。

### 活動共有ノートの解説ルール（今後この方針を固定）
以前の「家のゲーム機で全部を統一」は**廃止**した（NASを"独立したゲーム機"と呼ぶ、PCの"手元のセーブ"を出すなど、無理が出て逆に分かりにくくなったため）。代わりに次の方針で書く。

1. **NAS＝マウス・キーボード・モニターの無いPC（留守番PC）** を出発点にする。まず正体を正確に、やさしい日本語で言い切る。
2. 用語は出てきた順に **「◯◯とは、かんたんに言うと〜」** を一文添える。読者に用語の暗記を求めない。
3. たとえ（軽い比喩）は**成立する範囲だけで使い、破綻したら即やめる**。1つの世界観に全概念を押し込まない。
4. 各用語の推奨・言い換え（下は"縛り"ではなく初期値。合わなければ差し替え可）：

| 本物 | やさしい言い換え（初期値） |
|---|---|
| サーバー/NAS | **画面もキーボードも無いPC**（＝ずっと起動できる“留守番PC”） |
| その上で動くAI/プログラム | NASの中で動く**AI（ソフト）** |
| 操作する側のPC・スマホ | **手元のパソコン・スマホ**（遠隔で操作する側） |
| SSH等の遠隔接続 | 離れたPCを**遠隔操作する接続**（切ると中の作業も終わる、が基本） |
| tmux/常駐の仕組み | NASの中に作る**“閉じても消えない作業部屋”**（接続を切っても作業が続く） |
| 作業フォルダ/保存対象 | **作業フォルダ**（そのまま。“セーブ”等に置き換えない） |
| 同期ツール（Syncthing等） | 2台のPCのフォルダを**つねに同じ中身にそろえる仕組み** |
| ポート（番号付き接続口） | 番号ごとに役割が違う**接続口（番号つきの入り口）** |
| Docker/コンテナ | 本体を汚さず**隔離した小部屋でアプリを動かす仕組み** |

- 核心の設計思想は「**操作する接続**」と「**作業そのもの**」を分けること（tmuxの本質）。ここは比喩よりこの一文で伝える。
- 公開前に機微情報を**伏字／マスク／クリーン図解**にする（第1節のパターン）。
- **実体のない社会的指標**（フォロワー・スキ・フォロー数等）や**アカウント機能**を載せない（＝正直版）。
- 端末画面は「コマンド付きの再現ブロック」、GUIは「再現カード」。本物のスクショは使わない（＝クリーン再現）。

## 3. 関所の動作原則

1. **絶対に途中で止めない。** 判断に迷うものは安全側に倒して隠す。
2. **既存の Notion→GitHub→Pages の仕組みを壊さない。** 公開直前に検査ステップを1つ足すだけ。
3. **自動修復してそのまま公開。** 修復ログは `docs/security/publish-check-log.md` に追記（公開はログ完了を待たない）。
4. 実装は GitHub Actions を想定（ビルド後・デプロイ前のジョブ）。

## 4. 実装状況

- ルール正本：本ファイル（作成済み）。
- 既存の一次防御：`scripts/notion-sync.mjs` の `FORBIDDEN_PATTERNS`（検出時スキップ）。
- 関所（自動修復＋ログ）：GitHub Actions ジョブとして追加予定（本ルールを参照）。
